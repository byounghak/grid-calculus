**Phase 13 — Vector and tensor fields is done.** Version bumped to `0.13.0`. Generalizes `Field<T>` beyond the rank-0 / rank-1 cases shipped through Phase 2 by adding a rank-2 instantiation (`Field<core::Mat3d>`, where `Mat3d` is a new alias for `Eigen::Matrix3d` in `core/EigenAliases.hpp`), the rank-raising `diff::gradient(Field<Vec3d>) -> Field<Mat3d>` overload (convention `M(i, j) = ∂_j v_i`, the velocity-gradient form so the trace gives the divergence), the rank-lowering `diff::divergence(Field<Mat3d>) -> Field<Vec3d>` overload (convention `(div M)_i = ∂_j M(i, j)`, matched so that `divergence(gradient(v))` is the vector Laplacian by construction), and a new `tensor/Contraction.hpp` header with three pointwise primitives in `gridcalc::tensor`: `trace(Field<Mat3d>) -> Field<double>`, `singleContract(Field<Mat3d>, Field<Mat3d>) -> Field<Mat3d>` (matrix product), and `doubleContract(Field<Mat3d>, Field<Mat3d>) -> Field<double>` (full contraction `A:B = A(i,j)B(i,j)`, the natural pairing for the Phase 15 elastic-energy `½σ:ε`). All contractions are fully-materialized; expression templates remain deferred to Phase 15 alongside the func::evaluate-over-tensor-fields surface that motivates them. The new gradient and divergence overloads inherit Phase 7's `Order` template parameter (default `2`, `4` available). Roadmap acceptance bar discharged via three programmatic identity tests in CI: `HessianIsSymmetric` (the rank-raising restatement of `∇×∇φ = 0`; FD operators commute on a periodic uniform grid, so the test passes to round-off), `TraceOfGradientEqualsDivergence` (round-off agreement modulo summation order), and `DivergenceOfGradientEqualsLaplacian` (compared against the analytical vector Laplacian `-2 v` for a manufactured trig field, since the discrete `divergence(gradient(·))` is a stride-2 stencil per the standing Phase 2 STATUS-line phenomenon — converges at order 2 with a different leading-error constant than `diff::laplacian`'s 3-point stencil); plus per-operator order-2 convergence sweeps for the new gradient and divergence overloads on a manufactured rank-2 trig field whose every row has a non-trivial divergence (an early-iteration bug where one row had identically-zero analytical divergence is documented in the source comment so the trap doesn't get re-stepped). **Previously (Phase 11):** Functional with up-to-4th-order gradients. Version bumped to `0.11.0`. Extends `gridcalc::func::evaluate` with a fourth SFINAE-detected callable arity — `f(double psi, const Vec3d& grad_psi, double lap_psi, double biharm_psi)` — added at the head of the existing `if constexpr` dispatch ladder so the longest-arity branch wins under the existing tie-breaking rule. Five new tests in `test/pfc_functional_test.cpp`: `HandComputedAtN64` (acceptance), `SecondOrderConvergence`, `PsiOnlyArityStillWorks` and `GinzburgLandauStillWorks` (regression checks for the 1-arg and 3-arg branches under the new longest-arity ordering), and `SpectralCrossCheck` (assembles the PFC integrand from `spectral::laplacian` + `spectral::biharmonic` and asserts machine-precision agreement with the analytical reference, gated by `GRIDCALC_USE_FFT`). **Previously (Phase 8):** Higher-order spatial derivatives (3rd, 4th). Version bumped to `0.8.0`. Ships 31 named partial-derivative functions covering every distinct multi-index $\boldsymbol\alpha = (n_x, n_y, n_z)$ with $2 \leq |\boldsymbol\alpha| \leq 4$ on a periodic Cartesian-orthogonal grid (6 rank-2, 10 rank-3, 15 rank-4), plus `gridcalc::diff::biharmonic<Order>` for $\nabla^4$ with the `gridcalc::diff::d4<Order>` alias. Naming convention is `d<rank>d<axis-with-exponent>...` with lexicographic axis order and exponent `1` omitted (e.g., `d3dx2dy<Order=2>(psi)` is $\partial^3\psi/\partial x^2\partial y$). Each operator is templated on `<int Order = 2>` mirroring Phase 7, with `Order = 4` shipped. Internal: new sibling stencil tables `stencil::ThirdDerivative<Order>` and `stencil::FourthDerivative<Order>` (orders 2 and 4 each); new tensor-product helper `diff::detail::mixedDerivative<Nx, Ny, Nz, Order>` is the single implementation site for all 31 named functions. Biharmonic uses a fused six-term direct stencil ($\sum_a \partial_a^4 + 2\sum_{a<b} \partial_a^2 \partial_b^2$), not `laplacian(laplacian(psi))`, to avoid the stride-2 leading-error drift documented for `divergence(gradient(psi))` in Phase 2. Test surface: 4 weight-table audits + 62 convergence sweeps (31 partials × {Order 2, 4}) + 2 biharmonic convergence sweeps + 4 explicit $|\mathbf{k}|^4$ acceptance tests + 2 alias-parity tests = **74 new tests**, all passing. All 60 prior tests still pass without modification. `specs/roadmap.md` Phase 8 entry was edited in the same branch to match the expanded scope.

## Decisions worth knowing (encoded in the specs but easy to miss)

- **Gradients are reported in Cartesian** regardless of lattice basis. Lattice geometry is an implementation detail of `Grid`. (Locked in Phase 16.)
- **Unified mixing-by-default differentiation on multi-atom-basis grids.** At a site of sublattice α, gradients pull contributions from all nearby sites regardless of sublattice. The decoupled per-sublattice mode is opt-in via a policy tag. Weights are derived per `(α, derivative, accuracy)` by generalized finite differences (Taylor-moment matching). (Phase 17.)
- **Functional API for high-rank data** must use contraction expression templates (Phases 11 and 14) to avoid materializing rank-6 intermediate tensors (3⁶ = 729 components/point for a 4th-order gradient of rank-2 data).
- **FFT is verification only** — never the production differentiation engine, never a solver. PocketFFT, header-only, behind `-DGRIDCALC_USE_FFT=ON` (default on). The CI cross-check between FD and FFT operators is a permanent gate from Phase 9 onward.
- **Documentation toolchain split.** User Guide = `memoir` class (typography); Developer Note = `book` class because `doxygen.sty` conflicts with `memoir` on `\hangpara` / `\hangparas` and several preamble packages. `pdflatex` is locked. **Sphinx was dropped** when the repo went private — PDFs are now the only narrative-documentation output, and the "Hello grid" content moved into the User Guide as a chapter (was originally planned as a Sphinx page).
- **Repository visibility: private; license: proprietary.** No open-source license; redistribution requires authorization. `LICENSE` file added in Phase 0 will be an "All rights reserved" notice. There is no public docs hosting; PDFs ship as private-repo release assets only. Mission target ("production / industrial") is unchanged.
- **Phase 0 ships Linux-only CI.** The original roadmap acceptance bar for Phase 0 listed all four compiler families (GCC, Clang, Apple Clang, MSVC). Decided in the Phase 0 spec round to narrow that to Ubuntu GCC + Ubuntu Clang for cheaper iteration, and to insert a new **Phase 22 — Cross-platform CI hardening** before the v1.0 release phase (which moved to Phase 23). The full four-family compiler-support matrix in `tech-stack.md` is unchanged — it just isn't enforced in CI until Phase 22. Known accumulated-portability-bug risk recorded in `roadmap.md` "Risks to watch."
- **`byounghak/atomic-structure` integration deferred to Phase 16+.** Considered importing the existing private library [`atomic-structure`](https://github.com/byounghak/atomic-structure) (v1.6.0; locally at `/Users/byounghak/Projects/KMC refactor/Atomic structure`) as the backing implementation of Phase 1's `Grid`. Decided against: Phase 1 is Cartesian-orthogonal only and a 30-line in-house struct is the right weight; atomic-structure's `Box` + `Crystal` + `Structure` map naturally onto **Phase 16 (non-orthogonal Bravais)** and **Phase 17 (multi-atom basis)** instead. Integration approach (replace `Grid` internals vs. consume `Structure` directly) is decided in the Phase 16 spec round. Recorded as an Implementation note under both Phase 16 and Phase 17 deliverables in `roadmap.md`.
- **`Field<T, Policy = IndexPolicy::Periodic>` is templated on policy from Phase 1.** Only `Periodic` exists today; `Dirichlet` and `Neumann` arrive at Phase 19. The policy template parameter was the *one* piece of forward-looking design admitted at Phase 1 specifically so that Phase 19 lands as an additive change rather than a public-API break. Everything else in Phase 1 stays YAGNI.
- **`Field<T>` storage layout is i-fastest** (`linear_index = i + Nx * (j + Ny * k)`) and is part of the public contract from Phase 1 forward. Matches FFT libraries (PocketFFT, FFTW) and Eigen's column-major default. Phase 9 spectral verification and Phase 21 benchmarks both depend on this; a layout flip later would invalidate Phase-21 baselines.
- **`stencil::Coefficients<Order>` is a template from Phase 1**, with `Order = 2` specialized and `Order = 4` deferred to Phase 7. The primary template is intentionally undefined so unsupported orders trip at compile time at the call site rather than silently zeroing the stencil. Phase 7 adds a specialization with no call-site refactor.
- **First-derivative coefficients live in a sibling template `stencil::FirstDerivative<Order>` (Phase 2), not in `Coefficients<Order>`.** Reusing the `Coefficients` name for first-derivative weights would silently change the semantics of Phase 1's existing call sites (Laplacian uses second-derivative coefficients). Two narrowly-named tables alongside each other was the clean call. A unifying redesign — for example a 2-parameter `Coefficients<Derivative, Order>` — is deferred to Phase 7, when both tables gain `Order = 4` specializations and the trade-offs are easier to evaluate.
- **`core/EigenAliases.hpp` is the home for project-local Eigen typedefs.** Created in Phase 2 when `gradient` and `divergence` joined the Phase 1 `Grid` as call sites for `Vec3d`. Phase 1's `Vec3d` was relocated here without changing its fully-qualified name (`gridcalc::core::Vec3d`). Future tensor/matrix aliases (`Mat3d` etc., as needed by Phase 13/14) belong here as well.
- **`divergence(gradient(ψ))` is *not* identically equal to `laplacian(ψ)` at finite `h`.** The composed operator is effectively a stride-2 second-derivative stencil (`(f(i+2) - 2 f(i) + f(i-2)) / (4 h²)`) whereas `laplacian` uses the 3-point stencil `(f(i-1) - 2 f(i) + f(i+1)) / h²`. Both converge to `∇²` at order 2 but with different leading constants; Phase 2's round-trip test therefore compares to the analytical Laplacian, not to `laplacian()`. This is decided behavior, not a bug.
- **Eigen propagated as a `SYSTEM` include.** `gridcalc`'s `INTERFACE_INCLUDE_DIRECTORIES` for Eigen are added with the `SYSTEM` keyword in the root `CMakeLists.txt`, so `-Wconversion` (and the rest of our strict warning set) stays active on our code but does not fire on `<Eigen/...>`. Without this, `WARNINGS_AS_ERRORS=ON` in CI breaks on Eigen's NEON typecasting headers.
- **Code style.** Leading-underscore member vars (`_lattice`, never `_Capital` or `__double`); camelCase verb-prefixed methods (`getX`, `isY`, `findZ`, `buildW`); snake_case struct fields; PascalCase types; STL-protocol carve-out (`begin`/`end`/`size`/`empty`/`data`/`swap` keep canonical names as 1-line shims that delegate to the verb-prefixed accessor).
- **Compiler tier.** GCC, Clang, Apple Clang, MSVC are all blocking-CI tier-1. The "Windows best-effort" line in `tech-stack.md` is a support-priority statement, not a CI demotion.
- **Lightweight pandoc-based PDF render is in CI (post-Phase-4 interim).** Decided after Phase 4 merged. `scripts/render-docs.sh` + the CI `render-docs` job render the markdown notes to two aggregate PDFs (`user-guide.pdf`, `developer-note.pdf`) on every PR/push, uploaded as workflow artifacts. The toolchain is intentionally narrower than Phase 10's plan (`pandoc` + `pdflatex`, no `memoir`/`book`, no Doxygen integration). Phase 10 retires this script and CI job in favor of the full LaTeX skeleton; until then, the lightweight PDFs are the canonical reader-facing artifact.
- **Implicit schemes (Backward Euler, Crank–Nicolson) stay on Phase 20's schedule.** Considered pulling them forward after Phase 6 merged, weighing two alternatives: (a) FFT-diagonalization solver, periodic-only, ~days of work, leverages PocketFFT pin from `tech-stack.md`; (b) matrix-free Conjugate Gradient with a `LinearOperator` abstraction, periodic-or-not, ~week of work, sets up infrastructure that Phase 20 will need anyway for non-periodic BCs. Decided to defer — keep the original roadmap order, do Phase 7 (4th-order accuracy stencils) next. Phase 20 will introduce implicit schemes from scratch when Phase 19 lands non-periodic boundary policies; option (b) is the natural starting point at that point. Recorded so the trade-off doesn't get re-litigated when Phase 20 starts.
- **Per-phase User Guide / Developer Note notes from Phase 1 forward.** Decided after Phase 2 merged (rule applied from Phase 3 onward at first); extended to cover Phases 1 and 2 retroactively after Phase 4 merged, when the lightweight pandoc PDF render landed and the user wanted Phases 1 and 2 included in the resulting PDFs. Every feature PR from Phase 1 forward has a paired markdown note under `docs/user-guide/notes/phase-N-<slug>.md` (user-facing API summary + worked example) and `docs/developer-note/notes/phase-N-<slug>.md`. The Developer Note follows a fixed five-section structure (**Theory · Math derivation · Algorithm · Design decisions · References**) per `specs/CLAUDE.md` step 4d, with at least one external citation per note (DOI URL, full bibliographic entry, or permanent URL). `validation.md` carries an explicit checkbox enforcing both the structure and the non-empty References section. Phase 10 absorbs the accumulated notes into the LaTeX chapters and empties the two `notes/` directories.
- **Phase 8's d-prefix naming convention for partial-derivative operators.** Each partial derivative function is named `d<rank>d<axis-with-exponent>...` with **lexicographic axis order** (`x` → `y` → `z`) and exponent `1` omitted. Examples: `d3dx2dy<Order>(psi)` is $\partial^3\psi/\partial x^2 \partial y$; `d4dx2dy2<Order>(psi)` is $\partial^4\psi/\partial x^2 \partial y^2$. Rank-2 partials use the same scheme (`dxx`, `dyy`, `dzz`, `dxy`, `dxz`, `dyz`) — not the literal `mixedPartial(psi, i, j)` runtime-axis form from the original roadmap wording. Locked in Phase 8; future ranks (e.g., Phase 11's up-to-4th-order *gradient*, where the input is rank-1 vector-valued data) extend the convention naturally. The decoration `d<rank>` with no axis suffix denotes the **contracted scalar operator** (`d4` = $\nabla^4$); only added at Phase 8 for biharmonic, since `laplacian` already names $\nabla^2$ and isn't being renamed.
- **Sibling templates over a unified `Derivative<Rank, Order>` (second deferral, Phase 8).** Same call as Phase 2 made for `FirstDerivative<Order>`: rather than redesigning `stencil::Coefficients<Order>` (= 2nd derivative since Phase 1) into a 2-parameter `Derivative<Rank, Order>`, Phase 8 adds new sibling templates `stencil::ThirdDerivative<Order>` and `stencil::FourthDerivative<Order>`. Renaming the existing tables would be a public-API break; adding siblings is additive. The unification meta-decision is deferred again — without a use case where `Rank` is itself a meta-template parameter, the redesign cost exceeds its benefit.
- **Direct stencils for the biharmonic, not `laplacian∘laplacian`.** Phase 8's `biharmonic<Order>(psi)` is implemented as a fused six-term direct stencil ($\sum_a \partial_a^4 + 2\sum_{a<b} \partial_a^2 \partial_b^2$) — not via `laplacian(laplacian(psi))`. The composition would converge at the right order but with a different leading-error constant, exactly the stride-2 phenomenon already documented for `divergence(gradient(psi))` in Phase 2. The direct stencil sidesteps that drift and lands the roadmap acceptance bar ($\nabla^4\psi$ recovers $|\mathbf{k}|^4$) at the published constant.
- **PocketFFT is vendored, not `FetchContent`-fetched (Phase 9).** `pocketfft_hdronly.h` is committed under `extern/pocketfft/` (BSD-3-Clause); the upstream revision is recorded in `extern/pocketfft/VERSION.txt` (named `VERSION.txt`, not `VERSION`, because a bare `VERSION` in a SYSTEM include path shadows the C++ standard `<version>` header on case-insensitive filesystems — the build broke loudly on macOS the first time around). Decided in the Phase 9 spec round: build determinism without network access matters more for the project's "production / industrial" target than the upgrade-by-one-line convenience of `FetchContent`. `tech-stack.md`'s Dependencies section was updated to match; the prior wording said `FetchContent` and `MIT`, both incorrect.
- **`gridcalc::spectral::*` is verification-only, not a production differentiation engine (Phase 9).** Documented in every spectral header's file-level Doxygen block, in the User Guide note, and in `mission.md` / `tech-stack.md` / `roadmap.md`. Production differentiation stays in `gridcalc::diff`. The hybrid spectral API ships `spectral::partial<Nx, Ny, Nz>` as the generic operator + named wrappers `spectral::laplacian`, `spectral::biharmonic`, `spectral::gradient` — *not* a 1:1 d-prefix family mirroring Phase 8 (since the test fixture instantiates the right multi-index from the FD operator's tag).
- **FD–FFT cross-check is a permanent CI gate from Phase 9 forward.** [`test/fd_fft_crosscheck_test.cpp`](specs/../test/fd_fft_crosscheck_test.cpp) parameterizes over every Phase 1–8 FD operator at every advertised accuracy order (70 tests as of 0.9.0). Future phases that add new periodic FD operators must extend the fixture in the same PR; the gate catches accidental accuracy regressions automatically. Sweep grid: $N \in \{16, 32, 64, 128\}$. Gate's runtime budget is ~120 s on Ubuntu Clang.
- **Nyquist mode is zeroed for odd-rank derivative axes** in `spectral::partial` (Phase 9). The convention is encoded once in the helper, not duplicated per operator. For even-rank axes the multiplier $(ik)^N$ is sign-independent at $k = \pm\pi/h$ so no zeroing is needed. This is the standard remedy for the $\pm\pi/h$ aliasing of the Nyquist mode under odd-rank spectral differentiation.
- **Full LaTeX documentation toolchain landed at Phase 10.** The `memoir`-class User Guide and `book`-class Developer Note are built via `latexmk`-driven `pdflatex` from `docs/{user-guide,developer-note}/main.tex`. The Developer Note's API-reference appendix is auto-generated by Doxygen and `\input{}`'d via `docs/extract-api-reference.sh` (which strips the standalone-document framing from `refman.tex`). Doxygen runs over `include/gridcalc/` (excluding `detail/`). The lightweight pandoc render (`scripts/render-docs.sh` + the CI `Render Doc PDFs` job) is retired; pandoc dropped from the project's required toolchain. Local-render entry point: `scripts/build-docs.sh`. Build flag `GRIDCALC_BUILD_DOCS=ON` (default `OFF` so contributors without TeX Live aren't blocked by configure-time errors). The 18 per-phase markdown notes are absorbed into LaTeX chapters; from Phase 11 onward, doc deliverables go straight to the LaTeX sources.
- **`\since`-tag enforcement is a permanent CI gate (Phase 10 onward).** [`scripts/check-since.py`](specs/../scripts/check-since.py) regex-walks every public header (excluding `detail/`) and fails the `docs-lint` CI job if any public class / function / variable / type alias has a Doxygen block but no `\since` tag. Doxygen's `WARN_IF_UNDOCUMENTED=YES` (paired in the same job) catches public symbols with no Doxygen block at all. Math-mode "unknown command" warnings (the `\partial`, `\nabla`, etc. inside `$...$` blocks the project's headers use) are explicitly allowlisted in the lint job's grep step.
- **Phase 11 ships the contracted-scalar 4th-derivative argument only.** `func::evaluate`'s new 4-arg arity passes $\nabla^4\psi$ to the user callable as a `double` (the value of `diff::biharmonic(psi)` at the grid point), not as a rank-4 tensor and not as a packed `std::array` of unique multi-index components. PFC, the roadmap's worked example, only needs the contracted scalars (Laplacian and biharmonic). $\nabla^3\psi$ is not exposed at all in Phase 11: it has no useful scalar contraction (the natural rank-1 form $\nabla(\nabla^2\psi)$ is a `Vec3d` PFC does not exercise) and rank-3 / rank-4 tensor-valued callable arguments belong in Phase 13/14 alongside the tensor type they require. Sibling consequence: the dispatch ladder gains exactly one new branch (`f(double, const Vec3d&, double, double)`) at the head, picked under the longest-arity-wins rule.
- **Contraction expression templates deferred from Phase 11 to Phase 15.** The roadmap "Decisions worth knowing" line previously paired Phase 11 and Phase 15 for the contraction-ET work. Reviewed in the Phase 11 spec round and decided: with the contracted-scalar-only argument shape (above), no rank-3 / rank-4 intermediate tensor is ever materialized in Phase 11, so the rank-6 functional-eval explosion the ET layer is meant to fend off does not yet appear. The ET design choices are driven by the rank-2-and-higher field arguments arriving in Phase 13/15, where they belong.
- **`examples/` lives under a build-flag-gated subtree (Phase 12 onward).** New top-level option `GRIDCALC_BUILD_EXAMPLES` (default `OFF`) gates `add_subdirectory(examples)` from the root `CMakeLists.txt`. Mirrors Phase 10's `GRIDCALC_BUILD_DOCS` pattern: contributors without an interest in the demo binaries (or without the matplotlib stack used to render the figures) configure normally and see no examples-related cost. Both Ubuntu CI jobs configure with `-DGRIDCALC_BUILD_EXAMPLES=ON` so the demo source is type-checked under the strict warning set on every PR; CI never *runs* the binary — the runtime gate is the new acceptance test in `test/`.
- **Helpers under `examples/common/`, not `include/gridcalc/` (Phase 12).** The CH RHS, energy diagnostics, IC generator, and VTK writer are not part of the library's public API. The Phase 10 `\since`-tag lint regex covers `include/gridcalc/` only; placing the example helpers there would either force them onto the public API surface or require a lint-regex carve-out for a non-library subdirectory. Tests `#include` the helpers via a `target_include_directories(gridcalc_tests PRIVATE ${CMAKE_SOURCE_DIR}/examples)` entry in `test/CMakeLists.txt`, so the Phase 12 acceptance test runs regardless of `GRIDCALC_BUILD_EXAMPLES`.
- **Cahn–Hilliard demo uses explicit RK4, not IMEX / FFT-diagonalized (Phase 12).** The "FFT verification only" rule in `mission.md` rules out the spectral-diagonalization integrator; matrix-free implicit schemes wait for Phase 20's solver-scaffolding work. Phase 12 ships the explicit-RK4 path with a documented $\Delta t \sim h^4 / (M\kappa)$ stability cost ($\Delta t_{\max} \approx 0.0193$ at the demo defaults). Production users running CH at higher resolutions will need Phase 20's IMEX path, which can be retrofitted into `examples/cahn_hilliard.cpp` as an `--integrator imex` flag at that point.
- **Visual + monotone-coarsening acceptance bar for Phase 12.** Two evidence streams: three committed PNG snapshots in the User Guide chapter (qualitative match to published CH images) plus a programmatic test in CI that asserts Lyapunov decay of $F(t)$, monotone coarsening of $\int|\nabla\psi|^2$ across the post-transient pair (≥ 3% drop, locally observed 4.83%–7.85% across 5 seeds), and machine-precision mass conservation. A quantitative $R(t)\sim t^{1/3}$ power-law fit was considered and rejected for this phase: at the affordable grid size the slope estimate is noisy enough that any threshold band wide enough to be robust would also be uninformative. May ship later as a separate non-CI scaling benchmark.
- **Legacy ASCII VTK `STRUCTURED_POINTS` for snapshot output (Phase 12).** Hand-written ~30-line writer in `examples/common/vtk_writer.hpp`; no new build dependency. Binary VTK and XML `.vti` were considered (5–10× smaller files / better ParaView UX respectively) and rejected as scope creep for the demo budget. Production users needing binary or XML VTK should integrate `vtkIO` directly. The i-fastest layout `Field<double>` uses matches the order VTK expects for `STRUCTURED_POINTS`, so the data block is a flat read of `field.data()` with no transposition.
- **`gridcalc::core::Mat3d = Eigen::Matrix3d` directly (Phase 13).** Standard Eigen module; the unsupported tensor module remains deferred to Phase 15 alongside rank-3 / rank-4 types and the contraction expression-template layer. The `TensorFixedSize` portability risk recorded in "Open / deferred items" stays deferred.
- **Velocity-gradient index convention `M(i, j) = ∂_j v_i` (Phase 13).** Row index = vector component, column index = derivative axis. Trace gives the divergence; the matched `(div M)_i = ∂_j M(i, j)` makes `divergence(gradient(v))` equal the vector Laplacian by construction. Documented on the new gradient overload's Doxygen block and in the User Guide chapter.
- **Function overloading on input type for rank-raising operators (Phase 13).** Same `diff::gradient` and `diff::divergence` names; new overloads on `Field<Vec3d>` / `Field<Mat3d>`. Both inherit Phase 7's `Order` template parameter. Phase 15 may add a unifying templated form when rank-3 inputs arrive; for Phase 13's two cases plain function overloading is the smallest delta against the Phase 2 surface.
- **Phase 13 ships fully-materialized contractions; ETs deferred to Phase 15.** Mirrors the Phase 11 deferral rationale: rank-2-only inputs do not trigger the rank-6 explosion that motivates expression templates. The ET design choices are driven by the rank-3 inputs that arrive in Phase 15 (gradient of a `Field<Mat3d>`) and by the func::evaluate-over-tensor-fields surface added at the same time.
- **No explicit `curl` operator (Phase 13).** The literal roadmap example `∇×∇φ = 0` is captured by the Hessian-symmetry test on `gradient(gradient(phi))`. An explicit `curl` (rank-1 → rank-1) is not added speculatively. If a use case appears later, the natural form is `curl(v) := antisymmetric_part(gradient(v))`.
